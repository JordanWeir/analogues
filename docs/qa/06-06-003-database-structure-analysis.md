# Database Structure Analysis

This review compares the schema currently created by `src/tasks/init_workspace.rs` with the product, pipeline, orchestration, system-map, and blackboard design docs.

## 1. Table By Table Analysis Of Current Intended Use

| Object | Current intended use | Drift or design notes |
| --- | --- | --- |
| `run_metadata` | Single-row workspace control record for ticker, run slug, paths, status, schema version, created timestamp, and financial fetch status. | This is the right anchor for a per-run SQLite workspace, but schema drift management is underpowered. `schema_version` records a number, while the docs expect migrations to become a workspace service responsibility. |
| `stock_info` | Single-row company profile and quote context used by report payload loading. | Useful, but too thin for deterministic identity and market-data custody. It lacks CIK, quote timestamp, quote source, fiscal year metadata, and a clear split between static company identity and volatile market quote data. |
| `sec_raw_facts` | Raw SEC Company Facts observations with taxonomy, concept, unit, period, filing metadata, numeric value, raw JSON, and fetch timestamp. | This is one of the strongest current tables and matches the docs' requirement to preserve broad facts rather than only headline fundamentals. Add stronger uniqueness and lineage only after deciding how to handle amended filings and repeated comparative-period disclosures. |
| `canonical_metric_definitions` | Product-level list of required normalized metrics such as revenue, EPS, cash, debt, shares, and cash flow, independent of a specific company. | The concept is correct. It currently represents definitions, not metric recipes. The docs imply future canonical metrics need calculation methods, candidate concepts, period semantics, confidence, and unavailable states. |
| `canonical_metric_mappings` | Company-specific mapping from canonical keys to SEC taxonomy concepts and units, with confidence, rationale, selector, and active flag. | Useful as a seed and audit layer, but it cannot represent component-based metrics such as total debt from current plus noncurrent debt, nor multiple period-specific mappings for one canonical metric. |
| `supporting_metric_selections` | Selected non-standard SEC concepts that are useful for scenario, narrative, or crux analysis. | Matches the product thesis that exotic concepts are valuable. It should eventually link to entries, signals, scenarios, and worker runs so a selected metric has provenance and can be invalidated. |
| `raw_fact_metric_catalog` view | Deterministic concept/unit rollup over `sec_raw_facts`, including counts, date coverage, filing recency, and min/max values. | This is a good read model for agents and SQL-first exploration. It is a view, so expensive usage may eventually need a materialized counterpart or indexed table. |
| `concept_catalog_entries` | Materialized concept catalog with period-shape counts, dominant period shape, usability, plot readiness, narrative tags, and coverage metadata. | This matches the pipeline plan. `narrative_tags` as JSON is acceptable early, but a join table will be easier once tags drive worker dispatch, filtering, or scenario-specific concept selection. |
| `concept_review_decisions` | Audit trail for canonical or concept review decisions, including confidence, rationale, warnings, and payload JSON. | This is directionally useful, but it overlaps with the planned `entries`, `worker_runs`, and `quality_gate_results` ledgers. Keep it only if it remains a narrow concept-review audit table, otherwise fold it into the blackboard model. |
| `fundamentals` | Compressed headline facts and market values used by report loading and older starter report logic. | This appears legacy. It loses period semantics, lineage, confidence, and derivation method, and recent QA already found it can be misleading. It should become a compatibility view or be removed after report code reads from stronger observations. |
| `fundamental_observations` | Period-aware normalized observations built from canonical mappings, derived series, market data, or other sources. | This is the better canonical fact surface. It still needs stronger lineage to raw facts or calculations, plus confidence and method fields for derived or component metrics. |
| `canonical_fundamental_observations` view | Convenience view over `fundamental_observations` where `canonical_key` is present. | Useful read model. It can become the basis for replacing `fundamentals` if companion views expose latest values by period type. |
| `data_quality_flags` | Deterministic or worker-generated issues about missing, stale, mismatched, sparse, or unsafe data. | Still useful as a data-ledger table. It needs actionable severities, source object references, status, owner/worker attribution, and integration with quality gates or signals. |
| `data_gaps` | Open missing-data records seeded by initialization or later analysis. | This duplicates the planned signal ledger. It can remain as a bootstrap table now, but should become a `signals` view or be replaced by first-class board signals. |
| `sources` | Source metadata for filings, media, analyst notes, company releases, and other citeable material. | Good start, but too shallow for source custody. The docs call for source documents, claims, citations, evidence spans, and inspectable coverage. |
| `claims` | Source-backed claims extracted from sources, with type, side, confidence, metric, and notes. | Useful, but it needs many-to-many citation links, evidence spans, relation to entries/scenarios/sections, status, and worker provenance. |
| `sections` | Report section registry and optional final body text keyed by section. | Works as artifact/readiness state. It should not be the primary home for research conclusions; final prose should be compiled from reviewed board state and obligations. |
| `content_blocks` | Structured report-ready blocks within sections, with type, title, body, source note, and JSON payload. | Good artifact projection table. It should be treated as render-ready output or cache, not as the durable source of truth for financial analysis or claims. |
| `content_block_metrics` | Metric rows attached to a content block for display. | Useful for report rendering. Long-term, metrics should be sourced from `fundamental_observations`, calculation records, or scenario outputs, with this table acting as a display projection. |
| `content_block_items` | List items attached to a content block for rendering bullets or row-like report content. | Useful for HTML/report shape. It duplicates some future entry/obligation content and should stay presentation-scoped. |
| `narrative_map` | Single-row summary of dominant, bull, bear, consensus, and counter-narrative framing. | Product-facing concept is valid, but as a single row it is too coarse for source custody, contradiction handling, refresh, and worker provenance. It should become a projection from entries and claims. |
| `narrative_map_items` | Ordered narrative items by type. | Similar to `narrative_map`: useful for display, but not enough as durable research memory. It overlaps with future `entries`, `claims`, and `content_blocks`. |
| `watch_items` | User-facing signals to monitor, with title, description, signal type, and optional source. | This is distinct from blackboard `signals`: these are report/user watchlist items, not worker dispatch requests. Keep it, but link watch items to scenarios, crux entries, source claims, and dependency records. |
| `historical_analogues` | Report-ready analogue rows with setup, lesson, caveat, ordering, and source. | Useful artifact table. It should also connect to entries, claims, scenarios, and calculations because analogues can influence assumptions and need source-backed caveats. |
| `scenario_assumptions` | Scenario headers with order, name, stance, probability, description, and assumption summary. | Matches the scenario ledger. It needs status, versioning, stale flags, and links to crux entries, claims, calculations, and worker runs for refresh-safe scenario evolution. |
| `scenario_crux_assumptions` | Per-scenario crux and assumption rows with impact and optional source. | Correct concept, but currently stores crux text rather than linking to durable crux entries. Add entry/source/calculation links before refresh workflows depend on it. |
| `scenario_sensitivities` | Free-text scenario sensitivity notes. | Too unstructured for the financial engine direction. Keep for prose if needed, but add structured sensitivity inputs when values, units, low/base/high ranges, and formulas matter. |
| `scenario_signals` | Confirming and breaking scenario-specific signals, optionally linked to watch items and sources. | Useful for scenario-conditioned output. It should not be confused with blackboard work-queue `signals`; rename or clearly model these as `scenario_watch_signals` if both tables exist. |
| `scenario_periods` | Period-level scenario assumptions for revenue, growth, shares, margins, EPS, valuation multiples, and blend weights. | Core table for scenario math. It currently mixes assumptions and derived values. The pipeline also expects scenario output payloads, so computed outputs should be persisted separately or clearly flagged. |
| `monte_carlo_config` | Single-row deterministic Monte Carlo configuration for iterations, seed, and bins. | Good current fit. Add version/method references if the simulation assumptions become more complex. |
| `monte_carlo_summary` | Single-row persisted distribution statistics and methodology for the latest rendered run. | Good artifact/calculation output. It needs linkage to scenario version, calculation run, and artifact generation so stale summaries can be detected. |
| `monte_carlo_histogram_bins` | Histogram bins from the latest Monte Carlo run. | Good display output. It currently assumes one active simulation result; future comparison or refresh history will need run IDs. |
| `monte_carlo_scenario_probabilities` | Diagnostics comparing input, normalized, sampled, and observed scenario probabilities. | Good validation table. Add run/version linkage with the Monte Carlo summary. |
| `artifacts` | Registry of rendered artifacts such as report HTML path, type, created timestamp, and notes. | Useful but too thin for v1. Artifacts need dependency links, source payload hash/version, renderer version, status, and invalidation metadata. |

Notably absent from the current schema: `entries`, `entry_relations`, blackboard `signals`, `obligations`, `worker_runs`, `quality_gate_results`, `board_snapshots`, source document custody, durable calculation query/run tables, dependency/invalidation edges, and a persisted `scenario_outputs` table. The absence of `scenario_outputs` appears intentional in the current code path: scenario outputs are calculated in memory during report generation, while Monte Carlo summary and histogram results are persisted.

## 2. Recommendations

**Add the missing blackboard core before adding more bespoke research tables.** The docs repeatedly identify these as the v0.5/v1 durable control plane: `entries`, `entry_relations`, `signals`, `obligations`, `worker_runs`, `quality_gate_results`, and `board_snapshots`. These tables should become the shared place for reviewed observations, analyses, calculations, contradictions, gaps, cruxes, worker dispatch, quality checks, and report obligations. Without them, new experiments keep creating narrow tables that cannot be queried, reviewed, superseded, or refreshed consistently.

**Seed bootstrap obligations and initial board signals during workspace initialization.** The system map expects initialization to create default obligations and initial signals, but the current seed path only creates workspace metadata, sections, canonical definitions, stock info, and Monte Carlo config. Add obligations for material citation coverage, scenario crux explanation, SEC lag disclosure, historical analogue caveats, and confirming/breaking scenario signals. Add initial board signals for missing source pack, empty narrative map, empty scenario assumptions, open data gaps, and required financial mechanics experiments.

**Add an explicit dependency and invalidation ledger.** The system map expects refreshes to mark derived observations, entries, calculations, scenarios, sections, and artifacts stale when source facts, claims, cruxes, or prices change. Add a table such as `dependency_edges` with `source_type`, `source_id`, `target_type`, `target_id`, `relationship`, `created_by`, and `created_at`. Add status/version fields to dependent records or centralize status in the dependency layer.

**Add durable calculation records.** The financial-engine docs call for persisted calculation questions, SQL templates or formulas, inputs, outputs, interpretations, and dispositions. Add `analysis_queries`, `analysis_query_inputs`, `analysis_query_runs`, and `analysis_query_outputs` or equivalent tables. Calculation outputs can then be promoted into `entries` while preserving reproducible SQL, input facts, units, assumptions, and rejection/promoted/background status.

**Add stronger source custody and citation tables.** Keep `sources`, but extend the evidence model with source documents or snapshots, evidence spans/chunks, filing accession/form metadata, content hashes, and citation join tables. Claims, entries, scenario cruxes, content blocks, analogues, and obligations should cite source refs through a consistent join table rather than only carrying `source_id` or `source_note` columns.

**Add schema migration tracking.** `run_metadata.schema_version` is not enough to manage drift once existing run databases need upgrades. Add a `schema_migrations` table with migration id, applied timestamp, checksum or app version, and notes. This should be owned by the future `WorkspaceStore` rather than by ad hoc task logic.

**Add company identity and market quote custody.** Extend `stock_info` or split it into `company_profile` and `market_quotes`. Persist CIK, exchange, sector, industry, fiscal year convention, quote timestamp, quote source, currency, market cap method, and any stale/missing flags. This will make report readiness and source limitations less dependent on prose notes.

**Add metric recipes and observation lineage.** Keep `canonical_metric_definitions` and `canonical_metric_mappings`, but add recipe-level support for direct concepts, component concepts, preferred units, period semantics, formula/method, confidence, and unavailable states. `fundamental_observations` should link to raw fact IDs, source rows, or calculation runs so derived values such as total debt, TTM revenue, free cash flow, or margins are auditable.

**Add scenario and Monte Carlo run versioning.** `scenario_periods`, `monte_carlo_summary`, histogram bins, and probability diagnostics currently represent the latest state. Add scenario version or calculation run IDs so outputs can be tied to the assumptions that generated them and invalidated when cruxes or probabilities change.

**Remove or deprecate `fundamentals` as a source-of-truth table.** It is too compressed and has already shown data-quality risk. Replace it with views over `fundamental_observations`, `market_quotes`, and calculation records. Keep a compatibility view named `fundamentals` only until report payload code no longer depends on the old shape.

**Remove `data_gaps` once first-class `signals` exist.** Data gaps are work requests, so they belong in the signal ledger with priority, status, origin, and addressed-by links. Deterministic quality facts can remain in `data_quality_flags`; open work should not live in a separate gap table long-term.

**Demote `narrative_map` and `narrative_map_items` to projections.** The report still needs narrative summaries, but source-backed narrative memory should live in `entries`, `claims`, and `entry_relations`. Keep these tables only as report compiler output or replace them with views/materialized projections.

**Consolidate `concept_review_decisions` with the blackboard audit model unless it stays narrowly scoped.** Concept review decisions can become `entries` created by `worker_runs` and checked by `quality_gate_results`. If the table remains, constrain it to concept-catalog review audit rather than letting it become a parallel worker-result ledger.

**Keep `sections`, `content_blocks`, `content_block_metrics`, and `content_block_items` as artifact projections, not research state.** These are useful for rendering, but they should be regenerated from reviewed entries, scenario outputs, calculations, claims, and obligations. Avoid adding domain logic that only exists inside content blocks.

**Clarify the two meanings of "signal."** `scenario_signals` are user-facing confirming/breaking indicators tied to scenario monitoring. Blackboard `signals` are worker-dispatch work items. Keep both concepts, but consider renaming `scenario_signals` to `scenario_watch_signals` or explicitly documenting the distinction in the schema.

**Normalize JSON fields only where query pressure justifies it.** `period_shape_counts`, `narrative_tags`, `payload`, and review payload JSON are reasonable during exploration. Promote tags, payload fields, and period-shape details into relational tables only when workers, quality gates, or refresh logic need to query them directly.

**Suggested migration order:** first add migration tracking, then add blackboard core tables, then seed default obligations and initial signals, then convert `data_gaps` and concept-review outputs into signals/entries, then add calculation and dependency records, then replace `fundamentals` with observation-backed views, then demote narrative and content tables to compiler projections.
