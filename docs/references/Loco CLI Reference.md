# Loco CLI Reference

This project uses the Loco CLI through `cargo loco`, which compiles and runs the app-specific CLI binary for this repository (`bot_orb_trader-cli` in the help output).

The commands below were derived by exploring the local help pages, starting with `cargo loco --help` and then drilling into the major subcommands.

## Global Usage

```sh
cargo loco [OPTIONS] <COMMAND>
```

Global options:

- `-e, --environment <ENVIRONMENT>`: choose the runtime environment. Default is `development`.
- `-h, --help`: print help.
- `-V, --version`: print version.

Top-level commands:

- `start`: start the app server, worker, or scheduler modes.
- `db`: create, migrate, reset, inspect, seed, and dump database state.
- `routes`: print all application routes.
- `middleware`: print configured middleware.
- `task`: run a custom task by name.
- `jobs`: manage queued jobs.
- `scheduler`: run or inspect scheduled jobs.
- `generate`: scaffold code and project assets.
- `doctor`: validate configuration.
- `version`: display app version.
- `watch`: restart the app on file changes.

## `start`

Usage:

```sh
cargo loco start [OPTIONS]
```

Important options:

- `-w, --worker [<WORKER>...]`: start worker mode. You can optionally provide tags to run only certain jobs.
- `-s, --server-and-worker`: run server and worker in the same process.
- `-a, --all`: run server, worker, and scheduler in the same process.
- `-b, --binding <BINDING>`: set bind address.
- `-p, --port <PORT>`: set port.
- `-n, --no-banner`: disable startup banner.

Typical uses:

```sh
cargo loco start
cargo loco start --worker
cargo loco start --all
```

## `db`

Usage:

```sh
cargo loco db <COMMAND>
```

Subcommands:

- `create`: create the database schema.
- `migrate`: run pending migrations.
- `down [STEPS]`: roll back migrations. Default is one step.
- `reset`: drop all tables and reapply migrations.
- `status`: show migration status.
- `entities`: generate SeaORM entity files from the current schema.
- `truncate`: truncate table data without dropping tables.
- `seed`: load fixtures or dump table data.
- `schema`: dump database schema.

### `db create`

```sh
cargo loco db create
```

Creates the schema for the current environment.

### `db migrate`

```sh
cargo loco db migrate
```

Applies all pending migrations.

### `db down`

```sh
cargo loco db down [STEPS]
```

Arguments:

- `[STEPS]`: number of migrations to roll back. Default is `1`.

Examples:

```sh
cargo loco db down
cargo loco db down 2
```

### `db reset`

```sh
cargo loco db reset
```

Drops all tables, then reapplies all migrations.

### `db status`

```sh
cargo loco db status
```

Shows migration status.

### `db entities`

```sh
cargo loco db entities
```

Generates entity `.rs` files from the current database schema.

This is especially useful after creating or changing migrations.

### `db truncate`

```sh
cargo loco db truncate
```

Truncates data in tables without dropping the schema.

### `db seed`

```sh
cargo loco db seed [OPTIONS]
```

Options:

- `-r, --reset`: clear existing data before seeding.
- `-d, --dump`: dump database tables to files instead of loading fixtures.
- `--dump-tables <DUMP_TABLES>`: dump only selected tables.
- `--from <FROM>`: folder containing seed files. Default is `src/fixtures`.

Examples:

```sh
cargo loco db seed
cargo loco db seed --reset
cargo loco db seed --dump
cargo loco db seed --dump --dump-tables users
cargo loco db seed --from src/fixtures
```

### `db schema`

```sh
cargo loco db schema
```

Dumps the database schema.

### Common database workflow

For a model-first or migration-first flow, the help pages suggest this sequence:

```sh
cargo loco generate migration CreatePosts title:string
cargo loco db migrate
cargo loco db entities
```

## `task`

Usage:

```sh
cargo loco task [NAME] [PARAMS]...
```

Arguments:

- `[NAME]`: task name or identifier.
- `[PARAMS]...`: task parameters, typically `key:value`.

Example:

```sh
cargo loco task my_task foo:bar baz:qux
```

This is the ad hoc execution entry point for custom tasks generated by Loco.

## `jobs`

Usage:

```sh
cargo loco jobs <COMMAND>
```

Subcommands:

- `cancel`: cancel jobs by name.
- `tidy`: delete completed or cancelled jobs.
- `purge`: delete old jobs, optionally filtered and optionally dumped first.
- `dump`: save jobs to files.
- `import`: import jobs from a file.
- `requeue`: move stale `processing` jobs back to queue state.

### `jobs cancel`

```sh
cargo loco jobs cancel --name <NAME>
```

Options:

- `--name <NAME>`: name of one or more jobs to cancel.

### `jobs tidy`

```sh
cargo loco jobs tidy
```

Deletes jobs in `completed` or `cancelled` status.

### `jobs purge`

```sh
cargo loco jobs purge [OPTIONS]
```

Options:

- `--max-age <MAX_AGE>`: maximum age in days before deletion. Default is `90`.
- `--status <STATUS>`: filter by job status. Allowed values: `queued`, `processing`, `completed`, `failed`, `cancelled`.
- `--dump <DUMP>`: dump jobs to a file before deleting them.

Examples:

```sh
cargo loco jobs purge
cargo loco jobs purge --status failed --max-age 30
```

### `jobs dump`

```sh
cargo loco jobs dump [OPTIONS]
```

Options:

- `--status <STATUS>`: filter exported jobs by status.
- `-f, --folder <FOLDER>`: output directory. Default is the current directory.

Example:

```sh
cargo loco jobs dump --status queued --folder tmp/jobs
```

### `jobs import`

```sh
cargo loco jobs import --file <FILE>
```

Options:

- `-f, --file <FILE>`: path to the job export file to import.

### `jobs requeue`

```sh
cargo loco jobs requeue [OPTIONS]
```

Options:

- `--from-age <FROM_AGE>`: requeue `processing` jobs older than this many minutes. Default is `0`.

Example:

```sh
cargo loco jobs requeue --from-age 15
```

## `scheduler`

Usage:

```sh
cargo loco scheduler [OPTIONS]
```

Options:

- `-n, --name <NAME>`: run one specific scheduled job by name.
- `-t, --tag <TAG>`: run jobs associated with a tag.
- `-c, --config <CONFIG_PATH>`: load a dedicated scheduler config file instead of the environment config.
- `-l, --list`: list configured scheduler jobs.

Examples:

```sh
cargo loco scheduler --list
cargo loco scheduler --name nightly_cleanup
cargo loco scheduler --tag billing
```

This is the main operational entry point for inspecting and running scheduled work.

## `generate`

Usage:

```sh
cargo loco generate <COMMAND>
```

Subcommands:

- `model`: generate a model and test logic.
- `migration`: generate a migration file.
- `scaffold`: generate model and controller scaffolding together.
- `controller`: generate a controller and tests.
- `task`: generate a task.
- `scheduler`: generate scheduler configuration.
- `worker`: generate a worker.
- `mailer`: generate a mailer.
- `data`: generate a data loader.
- `deployment`: generate deployment infrastructure.
- `override`: copy built-in templates so you can customize them locally.

### `generate model`

```sh
cargo loco generate model <NAME> [FIELDS]... [--without-tz]
```

Arguments:

- `<NAME>`: model name.
- `[FIELDS]...`: field definitions such as `title:string` or `hits:int`.

Options:

- `--without-tz`: skip `created_at` and `updated_at`.

The help examples show that fields can express:

- required values via `!`, such as `title:string!`
- references such as `director:references`
- explicit foreign key columns such as `award:references:prize_id`

Examples:

```sh
cargo loco generate model posts
cargo loco generate model posts title:string! content:text
cargo loco generate model movies long_title:string director:references award:references:prize_id
cargo loco generate model posts title:string content:text --without-tz
```

### `generate migration`

```sh
cargo loco generate migration <NAME> [FIELDS]... [--without-tz]
```

Arguments:

- `<NAME>`: migration name.
- `[FIELDS]...`: table fields.

Options:

- `--without-tz`: skip timestamp columns.

The help output documents several naming patterns:

- `CreatePosts`: create a new table.
- `AddNameAndAgeToUsers`: add columns to an existing table.
- `RemoveNameAndAgeFromUsers`: remove columns from a table.
- `CreateJoinTableUsersAndGroups`: create a join table.
- any other freeform name, such as `FixUsersTable`, for a blank migration.

Examples:

```sh
cargo loco generate migration CreatePosts title:string
cargo loco generate migration AddNameAndAgeToUsers name:string age:int
cargo loco generate migration RemoveNameAndAgeFromUsers name:string age:int
cargo loco generate migration AddUserRefToPosts user:references
cargo loco generate migration CreateJoinTableUsersAndGroups count:int
cargo loco generate migration FixUsersTable
```

Suggested follow-up commands from the help page:

```sh
cargo loco db migrate
cargo loco db entities
```

### `generate scaffold`

```sh
cargo loco generate scaffold <NAME> [FIELDS]... [OPTIONS]
```

Options:

- `--without-tz`: skip timestamp columns.
- `-k, --kind <KIND>`: scaffold kind. Allowed values: `api`, `html`, `htmx`.
- `--api`, `--html`, `--htmx`: shorthand selectors for scaffold kind.

Examples:

```sh
cargo loco generate scaffold posts title:string! user:references --api
cargo loco generate scaffold posts title:string! user:references --api --without-tz
```

### `generate controller`

```sh
cargo loco generate controller <NAME> [ACTIONS]... [OPTIONS]
```

Arguments:

- `<NAME>`: controller name.
- `[ACTIONS]...`: actions to generate.

Options:

- `-k, --kind <KIND>`: controller type. Allowed values: `api`, `html`, `htmx`.
- `--api`, `--html`, `--htmx`: shorthand selectors.

Examples:

```sh
cargo loco generate controller posts --api
cargo loco generate controller posts --api list remove update
```

### `generate task`

```sh
cargo loco generate task <NAME>
```

Generates a task that can later be run with `cargo loco task <NAME>`.

### `generate scheduler`

```sh
cargo loco generate scheduler
```

Generates a scheduler configuration template.

### `generate worker`

```sh
cargo loco generate worker <NAME>
```

Generates a worker for background job processing.

### `generate mailer`

```sh
cargo loco generate mailer <NAME>
```

Generates a mailer.

### `generate data`

```sh
cargo loco generate data <NAME>
```

Generates a data loader.

### `generate deployment`

```sh
cargo loco generate deployment <KIND>
```

Arguments:

- `<KIND>`: deployment target. Allowed values: `docker`, `shuttle`, `nginx`.

Examples:

```sh
cargo loco generate deployment docker
cargo loco generate deployment shuttle
cargo loco generate deployment nginx
```

### `generate override`

```sh
cargo loco generate override [TEMPLATE_PATH] [--info]
```

Arguments:

- `[TEMPLATE_PATH]`: a specific template file or template directory to copy locally.

Options:

- `--info`: show available templates under the specified directory without copying them.

Examples from the help page:

```sh
cargo loco generate override scaffold/api/controller.t
cargo loco generate override migration/add_columns.t
cargo loco generate override scaffold/htmx
cargo loco generate override task
cargo loco generate override .
```

Use this when you want to customize generated code templates instead of accepting the built-in defaults.

## Other Useful Commands

### `routes`

```sh
cargo loco routes
```

Prints all application endpoints.

### `middleware`

```sh
cargo loco middleware
```

Prints application middleware. The help page also shows a `-c, --config` flag, but it does not include a description in the CLI output.

### `doctor`

```sh
cargo loco doctor
```

Useful options:

- `-c, --config`: print the current configuration.
- `-p, --production`: available in help output, but without additional description.

### `version`

```sh
cargo loco version
```

Displays the app version.

### `watch`

```sh
cargo loco watch [OPTIONS]
```

Options:

- `-w, --worker [<WORKER>...]`: start worker mode while watching for changes.
- `-s, --server-and-worker`: run server and worker together in watch mode.

## Practical Shortcuts

Common development loops based on the discovered help pages:

```sh
# Generate a model or migration, then apply it
cargo loco generate model post title:string! body:text
cargo loco db migrate
cargo loco db entities

# Generate and run a task
cargo loco generate task sync_prices
cargo loco task sync_prices source:orb

# Inspect and run scheduled jobs
cargo loco scheduler --list
cargo loco scheduler --name nightly_cleanup

# Maintain the jobs table
cargo loco jobs dump --status failed --folder tmp/jobs
cargo loco jobs purge --status failed --max-age 30
```

## Notes From Exploration

- All top-level and nested commands consistently support `--environment`, `--help`, and `--version`.
- `generate`, `db`, and `jobs` are the largest command families and cover most Rails-like generator and maintenance workflows.
- Several help examples use the shorter alias form `cargo loco g ...`, even though the top-level help page spells the command out as `generate`.
- On the installed `loco-rs 0.16.4`, `cargo loco generate override . --info` panicked with `TemplateNotFound { path: "." }` during exploration, so use a concrete template path if you want to inspect overrides with `--info`.
